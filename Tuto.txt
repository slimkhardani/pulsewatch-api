##-----------------Step 1------------------
1)Database Architecture & Entity
User Entity:
This class represents the system users who manage the monitoring targets
 - id
 - email
 - passwordHash
 - plan : Restricts users to either 'free' or 'pro'
 - createdAt
 - monitors : Defines a 1-to-many link establishing that a single user account can own and manage multiple target monitors.

Monitor Entity:
This is the core configuration entity representing a single target URL or endpoint that needs to be systematically pinged.
 - id
 - name
 - url : The target endpoint address to check.
 - intervalSeconds : The frequency gap between pings, defaulting to 300 seconds (5 minutes).
 - isActive : Flags whether the background worker should actively monitor this URL or skip it.
 - user : Links back to the owning User. Crucially, { onDelete: 'CASCADE' } guarantees that if a user deletes their account, all associated monitors disappear automatically.
 - checks & incidents : Inverse relational definitions tracking historical logs and downtime events mapped to this configuration.

Check Entity:
This is a high-volume historical log table. Every single ping request made by our background engine logs a entry here, capturing metric tracking snapshots over time.
 - id
 - statusCode : The returned HTTP status (e.g., 200, 404, 500). It is marked nullable: true because a timeout or net failure won't return a status code.
 - responseTimeMs : Latency of the server round-trip in milliseconds, useful for performance graphs.
 - success : Quick true/false flag determining if the endpoint successfully responded within constraints.
 - checkedAt
 - @Index(['monitor', 'checkedAt']): A composite index strategy grouping the monitor reference and time. This ensures that querying performance graphs over specific date periods stays fast, even with millions of rows.

Incident Entity:
While checks track every single ping, incidents isolate states of critical downtime.
 - id
 - startedAt : Automatically stamps the exact moment the service started failing.
 - resolvedAt : Marked nullable: true while the issue is actively ongoing; stamped with a value only when the service recovers.
 - cause : Descriptive context regarding the underlying failure pattern (e.g., Timeout, 502 Bad Gateway).

2)Module Wiring Layer
src/app.module.ts
This acts as the root orchestrator of your application. It imports the structural environment variables via ConfigModule and initializes TypeOrmModule.forRoot(). By explicitly injecting our defined array of entities ([User, Monitor, Check, Incident]) and turning on synchronize: true, NestJS communicates the structural definition to Postgres directly at launch.

The role of synchronize: true is to act as an automatic bridge between your NestJS code and your database during development. Whenever you start your server, it reads your TypeScript entity files and automatically creates, modifies, or drops the actual PostgreSQL tables so they match your code exactly. This drastically speeds up your workflow because you don't have to manually write SQL commands as you build. However, it is strictly meant for local development; if you accidentally delete a property in your code, synchronize will instantly delete the corresponding column and all of its real data, which is why we turn it off and use careful, step-by-step "migrations" before deploying to production.



##-----------------Step 2------------------
1)Data Transfer Objects (DTOs)
register.dto.ts & login.dto.ts: These classes act as the validation gatekeepers for data entering the application. Combined with ValidationPipe inside main.ts, they inspect incoming request payloads before they ever hit your logic. If an email is improperly formatted or a password is too short, NestJS automatically rejects the request with a clean 400 Bad Request error.

2)Users Component Layer
 - users.service.ts: This is the internal data abstraction layer. It directly interacts with the PostgreSQL users table via the TypeORM repository wrapper. It provides straightforward methods to query users by their unique ID or email, and creates new database rows when users register.
 - users.module.ts: This encapsulates the user database layer. Crucially, it lists UsersService in its exports array. This makes the database operations accessible to the AuthModule without exposing the underlying configuration details.

3)Core Authentication Logic Layer
 - auth.service.ts: This handles your business validation rules. During registration, it safely encrypts plain passwords into strong hashes using bcrypt before database storage. During login, it compares provided passwords against database hashes. If matching, it passes user context into JwtService to sign an identity token.
 - auth.controller.ts: This maps network traffic directly to the service layer. It exposes public REST api endpoints (/auth/register and /auth/login) so client applications can send authentication payloads.

4)Passport & Token Protection Strategies
 - jwt.strategy.ts: This background passport configuration decodes incoming string tokens. It listens to incoming HTTP request headers, pulls the token from the standard Authorization: Bearer <token> location, verifies its signature against the secret password key (JWT_SECRET), and returns the payload data.
 - jwt-auth.guard.ts: A highly reusable decorator class. By placing this guard above any route controller method, you tell NestJS to block unauthenticated requests and automatically attach the verified user's information (req.user) straight into your route logic handler.
 - auth.module.ts: The central nervous system for authentication. It wires together security configurations, reads parameters such as expiresIn timers from environment variable files, and configures the token generation engine.

5)Global Boot Setup & Verification
 - app.module.ts: We integrated the UsersModule and AuthModule into our core infrastructure imports alongside the existing database connector.
 - main.ts (ValidationPipe): Passing { whitelist: true } globally means NestJS will strip out any unexpected or malicious fields sent in JSON payloads before running validation rules.

How the Token Verification Cycle Works
 1. Authentication: The client sends an email and password to /auth/login.
 2. Signing: If valid, the system signs a token containing the user's ID (sub) and hands it back.
 3. Authorized Requests: For protected routes, the client places this token inside an Authorization request header.
 4. Guard Interception: The JwtAuthGuard captures the request, verifies the signature using your JWT_SECRET, and lets the operation complete safely.



##-----------------Step 3------------------
In this step, we built the core Monitors CRUD (Create, Read, Update, Delete) engine. This layer allows authenticated users to manage the specific websites or APIs they want monitored. Crucially, we implemented tight multi-tenant security: a user can only interact with monitors they own, and any attempt to view or modify someone else's monitor will be rejected by the system.
1)The Validation & Inheritance Layer (DTOs)
 - create-monitor.dto.ts: Ensures incoming data contains a clear label (name) and a valid endpoint formatting schema (url). It also enforces a safety minimum limit on intervalSeconds (60 seconds) so users don't crash our system by demanding 1-second pings.
 - update-monitor.dto.ts: Uses @nestjs/mapped-types's PartialType. This maps directly to your CreateMonitorDto but automatically marks every field as optional. This allows clean, partial HTTP PATCH updates (e.g., updating just the name without re-sending the URL).
   When a user wants to update a monitor they have already created, they use an HTTP PATCH request. They usually don't want to change everything—they might just want to change the name, or just the url, or just the interval.
   If we used CreateMonitorDto, NestJS would crash and say: "Hey! You didn't provide the URL or the Name!" because CreateMonitorDto makes those fields required.
   This is where UpdateMonitorDto comes in. Its role is to tell NestJS: "Take all the validation rules from CreateMonitorDto, but make every single field optional."

2)The Custom Request Extracter (Decorator)
 - current-user.decorator.ts: The @CurrentUser() decorator abstracts the extraction of the authenticated user context from the execution lifecycle. During the execution of the JwtAuthGuard, Passport validates the bearer token, decodes the encrypted JSON payload, and appends the resulting identity object directly onto the Express request stream as request.user. Instead of forcing every controller method to inject the entire, heavyweight HTTP Request object and parse it manually via request.user, this custom parameter decorator intercepts the ExecutionContext, isolates that specific block of mutated request metadata, and injects exactly the { userId, email } payload directly into the method arguments.

3)Secured Endpoint Controller Layer
 - monitors.controller.ts: By wrapping the entire class in @UseGuards(JwtAuthGuard), all CRUD routes under /monitors are locked down instantly. The controller routes incoming endpoints directly into specialized user context actions.

4)The Data Isolation Isolation Layer (Service)
 - monitors.service.ts: Handles the heavy structural database manipulation and security validation:
   * Ownership Isolation: In findOneForUser, it explicitly extracts the relational mapping (relations: ['user']). It runs an explicit check: if (monitor.user.id !== userId) throw new ForbiddenException();. If a logged-in user tries to guess a monitor's UUID string in their browser, the server blocks them immediately.
   * Type Casting Structure: In create, the line user: { id: userId } as any sets up the reference relation cleanly in TypeORM without needing to execute an expensive database select call just to fetch the full user entity.

##########---------------------------------Week 2---------------------------------##########
What is Redis?
Unlike traditional databases (like PostgreSQL or MySQL) that write data to slow mechanical disks or solid-state drives, Redis keeps all its data directly in the computer’s (RAM).
This makes reading and writing data take less than a millisecond. 

What is BullMQ?
BullMQ is a fast, robust, production-ready queue library for Node.js/TypeScript. It does not replace a database or Redis; instead, it sits on top of Redis. BullMQ uses Redis's speed and atomic commands to create, manage, and distribute background jobs.

1)ping.util.ts
The simplest file — its only job is: "given a URL, visit it, and tell me if it worked, how long it took, and what went wrong if it didn't.

2)check.entity.ts
This defines what a single "checkup" looks like when saved to the database — like a medical chart entry. Every time we check a website, one row gets added here: did it succeed, what time, how fast.

3)checks.service.ts
The helper that actually saves those checkup rows to the database, and can fetch recent ones back. Nothing clever — just "save a check" and "get recent checks."

4)incident.entity.ts
Defines what an "incident" looks like — a period where a website was down. It has a start time and (once fixed) an end time. If the end time is empty, it means "still broken right now."

5)incidents.service.ts
The helper for incidents: "is there currently an open incident for this website?", "open a new incident", "close this incident because it's fixed now."

6)monitoring.module.ts
This is where we set up the "whiteboard" (Redis queue) itself and connect all the pieces (Checks, Incidents, Monitors) together so they can talk to each other.

7)monitoring.scheduler.ts
its job is purely to manage the to-do list (the queue). When a new website is added to watch, this file writes a ticket saying "check this one every 60 seconds, forever." When a website is deleted or paused, this file removes that ticket.

8)monitoring.worker.ts
this is the file that actually does the real work. It watches for tickets, and for each one:
 1-Looks up the website's URL
 2-Uses ping.util.ts to check if it's alive
 3-Saves the result using checks.service.ts
 4-If it failed twice in a row, opens an incident using incidents.service.ts; if it just recovered, closes the incident

9)monitors.service.ts
Before Week 2, this file could only save/list/delete websites in the database. Now, whenever you add, change, or remove a website, it also tells the scheduler ("waiter") to update the ticket board — so new websites actually start getting checked immediately instead of waiting for a restart.

**The Objective
Week 2 gave PulseWatch a working engine that silently detects downtime. Week 3 makes that detection actually useful to a human: get notified the moment something breaks, and see summarized health data instead of raw database rows.

**What We Built
Alerts — when the worker opens or resolves an incident, it now also sends a real email (via Resend) and optionally a Slack message, instead of just logging to the console.
Aggregation endpoints — three new API routes that turn raw Check rows into the kind of summarized numbers a dashboard would display: uptime percentage, average response time, and time-series data for chart

**File-by-File

1)alerts/alerts.service.ts
the only file that knows how to actually send an email or Slack message. Has three methods: sendDownAlert, sendRecoveryAlert, sendSlackAlert. Nothing about monitors or incidents — just "given this info, send this message."

2)alerts/alerts.module.ts
standard NestJS wiring, makes AlertsService available to other modules.

3)checks.service.ts (updated)
gained two new methods: getUptimeStats() (counts successes vs. total checks in a time window, computes a percentage and average response time) and getResponseTimeSeries() (returns raw check data points in chronological order, ready for a chart).

4)monitors.service.ts (updated findById) 
now fetches the related User row along with the monitor, because the worker needs the owner's email/Slack webhook to send alerts. Without this, monitor.user would be undefined and alerts would silently never fire.

5)monitor-stats.controller.ts (new) 
exposes the three new HTTP routes: /stats/uptime, /stats/response-time, /stats/recent-checks. Every route first calls findOneForUser() to confirm the requester actually owns the monitor before returning any data — same ownership pattern from Week 1's CRUD.


##########---------------------------------Week 4---------------------------------##########
**The Objective
Weeks 1-3 built a fully working backend, but everything lived in curl/Postman — invisible to an actual user. Week 4's job was to build a real React app so someone can log in, see their monitors, manage them, and view their health data visually. This is what turns PulseWatch from "a backend I can prove works" into "a product I can screenshot and demo."

**Architecture Decision: Separate Frontend Project
We created pulsewatch-frontend as a completely separate Vite + React project, not bolted onto the NestJS backend. This mirrors how real companies structure things — frontend and backend are independently deployable, independently scalable, and can even be built by different teams. The two talk to each other purely over HTTP (REST calls), never sharing code directly.

**File-by-File Explanation
1)src/api/client.js 
A single, shared Axios instance used by every page instead of each page configuring its own HTTP calls. Its one clever trick: an interceptor that automatically attaches the JWT token to every outgoing request's Authorization header, read fresh from localStorage each time. This means individual pages never have to think about "did I remember to send the token?" — it's handled once, centrally.

2)src/pages/Login.jsx / Register.jsx
Simple forms that call /auth/login or /auth/register, and on success, store the returned JWT in localStorage and redirect to /dashboard. localStorage was chosen because it's the simplest option that survives page refreshes (unlike component state, which resets on reload).

3)src/pages/Dashboard.jsx
The main landing page after login. Does three things: fetches and lists all the user's monitors (GET /monitors), provides a form to add a new one (POST /monitors), and — after Week 5 — shows the user's plan and monitor count. Each monitor in the list is a clickable card linking to its detail page.

4)src/pages/MonitorDetail.jsx
The "drill-down" page for one specific monitor. On load, it fires three parallel API calls: uptime stats, response-time series (for the chart), and recent checks (for the activity feed). These three calls map directly to the three aggregation endpoints we built in Week 3 — this page is literally the visual consumer of that backend work.

5)src/App.jsx
Defines all the routes (/login, /register, /dashboard, /monitors/:id) using React Router, and wraps the protected ones in a PrivateRoute component that checks for a token in localStorage before allowing access — if there's no token, it redirects straight to /login. This is the frontend's own lightweight "auth guard," separate from (but working alongside) the backend's JwtAuthGuard.



##########---------------------------------Week 5---------------------------------##########
**The Objective
Two separate upgrades bundled into one week: (1) make the dashboard update live the instant something happens, instead of requiring a manual refresh, and (2) turn the "free vs. pro" plan field from a decorative column into something real — actual payments that unlock actual limits.

Part A — Realtime with WebSockets
**Why This Was Needed
Without it, a user watching their dashboard would only see a new check or incident by refreshing the page. Real monitoring products (UptimeRobot, Better Uptime) update live — this is a genuine product-quality gap we closed.

**File-by-File
1)src/monitoring/monitoring.gateway.ts
A NestJS WebSocket Gateway, which is just NestJS's term for "a class that manages a live, two-way connection with connected browsers" (as opposed to a Controller, which only handles one-off request/response HTTP calls). This file is deliberately "dumb" — it has no idea what a ping or an incident even is. It exposes two methods, emitCheckResult() and emitIncidentUpdate(), that just take data and broadcast it to every connected client. The separation matters: the Gateway only knows how to send a message; the Worker decides when and what.

2)monitoring.worker.ts (updated)
At the exact two moments it already existed — right after saving a Check, and right after opening/resolving an Incident — we added one more line calling the Gateway to broadcast that same event live. Nothing about the core detection logic changed; we just added a "also tell anyone listening" step at the end.

3)src/api/socket.js (frontend)
A single shared Socket.io client instance, same pattern as client.js for REST calls. Configured with autoConnect: false so pages control exactly when to open/close the connection, rather than it connecting globally on app load.

4)MonitorDetail.jsx (updated)
On mount, connects the socket and subscribes to two event names scoped to that specific monitor's ID (monitor:${id}:check and monitor:${id}:incident). When a check event arrives, it prepends the new entry to the recent-checks list and appends a new point to the chart data — all without touching the REST API again. On unmount (leaving the page), it disconnects — good practice to avoid leaking open connections as the user navigates around.

Part B — Stripe Billing
**Why This Was Needed
A plan field existed on the User entity since Week 1, but nothing ever changed it or enforced it — anyone could create unlimited monitors regardless of "plan." This week made that distinction real: actual payment processing, actual enforcement.

**File-by-File
1)src/billing/billing.service.ts
  * createCheckoutSession() — builds a Stripe-hosted payment page URL for a specific user, tagging it with metadata: { userId } so we can identify who paid once Stripe confirms it
  * handleWebhookEvent() — processes events Stripe sends to us. We only act on two: checkout.session.completed (upgrade the user to pro) and customer.subscription.deleted (downgrade back to free). Every other event type Stripe sends (and it sends many — invoice created, payment succeeded, customer updated, etc.) is safely ignored, not treated as an error.

2)src/billing/billing.controller.ts
Exposes two routes: /billing/checkout (protected by JwtAuthGuard, since only a logged-in user should be able to start a checkout for themselves) and /billing/webhook (deliberately not protected by the JWT guard, because Stripe's servers call this directly — they don't have a user's JWT. Security here instead comes from cryptographically verifying Stripe's signature on each incoming request).

3)main.ts (rawBody: true)
Stripe's signature verification needs the exact, untouched bytes of the incoming request. NestJS's default behavior is to parse the JSON body before your code ever sees it, which would corrupt those exact bytes. rawBody: true preserves the original raw buffer specifically so handleWebhookEvent() can verify it correctly.

4)monitors.service.ts (create() updated)
Before creating a new monitor, we now look up the user's current plan fresh from the database and count their existing monitors, comparing against a { free: 5, pro: 50 } limits map. Looking it up fresh (not trusting a cached value) matters because a user's plan can change at any moment — if we trusted an old cached value, someone who just upgraded might still get incorrectly blocked.

5)Stripe CLI (stripe listen) 
A tool that solves a specific local-development problem: Stripe's servers live on the public internet and have no way to reach localhost:3000 on your laptop directly. stripe listen creates a temporary tunnel — Stripe sends real events to Stripe's own relay, which forwards them to your local backend. In production, this tool isn't needed at all — Stripe would call your real deployed URL directly.


##########---------------------------------Week 6---------------------------------##########
**The Objective
Close four real gaps left over from Weeks 1-5: no way to recover a forgotten password, no verification that signup emails were real, no way to actually cancel a paid plan, and dead code for a Slack integration that was never tested. This work turned PulseWatch from "feature-complete in the happy path" into "handles the edge cases a real product needs."

**Feature 1 — Slack Removal (cleanup)
Removed sendSlackAlert() from AlertsService, deleted the two slackWebhookUrl conditional blocks from monitoring.worker.ts, and dropped the slackWebhookUrl column from the User entity. Since synchronize: true is still active, TypeORM applied that schema change automatically on the next boot — no manual migration needed at this stage. This left AlertsService focused purely on email, which is what's actually tested and working.

**Feature 2 — Email Verification (Option A: hard block)
Added isEmailVerified and emailVerificationToken to the User entity. On register(), we generate a random token (crypto.randomBytes(32)), save it, and email a verification link — but critically, registration no longer issues a JWT. The user is created in an unverified state and cannot log in until they click that link. login() now explicitly checks isEmailVerified and throws a 401 with a clear message if not. A new /auth/verify-email endpoint takes the token, looks up the matching user, and flips isEmailVerified to true.
We also handled a real edge case: what happens if someone registers with an email that's already been used but never verified? Originally this hit a hard "email already in use" conflict with no way forward. We fixed register() to check verification status first — if the existing account is unverified, it silently regenerates a fresh token and resends the email instead of blocking the user with a dead end. A companion /auth/resend-verification endpoint (plus a "Resend verification email" link on the Login page) covers the case where the original email got lost or expired.

**Feature 3 — Password Reset
Added passwordResetToken and passwordResetExpires (a 1-hour expiry) to User. /auth/forgot-password generates a token, emails a reset link, and — deliberately — always returns the same generic "sent" response whether or not the email exists, so the endpoint can't be used to probe which emails are registered accounts. /auth/reset-password validates the token isn't expired, hashes the new password, and clears the reset fields. Frontend pages (ForgotPassword.jsx, ResetPassword.jsx) were added along with a "Forgot password?" link on the Login page.

**Feature 4 — Subscription Cancellation
Added stripeSubscriptionId to User, captured from the checkout.session.completed webhook payload (previously we only stored the customer ID, not the subscription ID — needed to actually target a cancellation). A new /billing/cancel endpoint calls stripe.subscriptions.cancel() directly against Stripe's API. Importantly, this endpoint doesn't downgrade the user itself — it triggers Stripe to fire the customer.subscription.deleted webhook, which flows back through the same webhook handler built in Week 5 to set the plan back to free. This was tested end-to-end with stripe listen running and confirmed working — the full upgrade → cancel → downgrade loop is proven, not just assumed.
